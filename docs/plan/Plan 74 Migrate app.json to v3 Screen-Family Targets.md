## Task Analysis
- Main objective: Convert `app.json` from the current Zepp OS `v2` device-specific manifest into a Zepp OS `v3` screen-family manifest while preserving the already-established Task 73 identity baseline (`appId` `1108585`, version `3.0.0`) and the current eight-page app surface.
- Identified dependencies: Task `73` is complete and already removed `app-side` / `setting`; current `app.json` still uses `configVersion: "v2"`, an `app` identity block, Zepp OS `1.0.x` runtime versions, legacy named targets (`gtr-3`, `gts-3`), and `device:os.local_storage`; `package.json` and `utils/version.js` currently align on `3.0.0`; `docs/PRD-Zepp OS API Level 3.6 Migration for New App.md` sets the support policy to API_LEVEL `3.6` with target families `w390-s`, `w454-r`, `w466-r`, and `w480-r`.
- System impact: High configuration/build impact but intentionally narrow code impact; the expected implementation is mostly confined to `app.json`, yet an invalid manifest would block Zepp builds, simulator loading, and all downstream migration tasks (`75`-`79`).

## Chosen Approach
- Proposed solution: Use a manifest-only migration flow centered on `app.json`: first lock the exact Zepp OS `v3` schema shape against official tooling/docs, then rewrite header/runtime fields, replace legacy device targets with the four required screen families, re-audit permissions and page registration, and only after that remove leftover `v2`-only properties.
- Justification for simplicity: This keeps Task `74` scoped to configuration rather than mixing in runtime API rewrites from later tasks, reuses the current page inventory and Task `73` decisions, and minimizes regression surface to one manifest file plus validation.
- Components to be modified/created: `app.json` and this plan file. Supporting references during implementation: `package.json`, `utils/version.js`, `.taskmaster/tasks/tasks.json`, `README.md`, and `docs/PRD-Zepp OS API Level 3.6 Migration for New App.md`.

## Implementation Steps
1. Establish the pre-edit baseline and schema lock for subtasks `74.1`-`74.5`: confirm current repo truth (`appId` `1108585`, version `3.0.0`, no `app-side` / `setting`, eight page files present), snapshot the current manifest, and treat the official Zepp OS `v3` validator/docs as the source of truth if the PRD (`targets.platforms` qualifiers) and Taskmaster detail (`screenFamily` example) differ in literal JSON shape.
2. Execute subtask `74.1` - update header/version configuration: change `configVersion` to `v3`, migrate the identity/runtime header into the v3-compliant structure, set the runtime/API minimum support policy to `3.6`, preserve the current `appId` and version baseline (`1108585` / `3.0.0`), and cross-check against `package.json` plus `utils/version.js` so release automation stays aligned. Implementation note discovered during schema lock: the working Zepp OS v3 sample manifest uses `targets.<key>.platforms[].dw` for per-platform design width, not v2-style top-level `designWidth`.
3. Execute subtask `74.2` - configure screen-family targets: replace `gtr-3` / `gts-3` and their `deviceSource`-based `platforms` blocks with explicit support for `w390-s`, `w454-r`, `w466-r`, and `w480-r`; keep the same page coverage for each supported family (or centralize shared page modules if the validated v3 schema supports it). Refined implementation choice: collapse to one generic v3 target key (`gt`) in the style of the official v3 samples, put all four screen families under that target, and keep per-platform `dw` values to preserve the existing 454-round / 390-square layout baseline while still removing legacy `deviceSource` and split target namespaces.
4. Execute subtask `74.3` - audit permissions: compare the current manifest permission (`device:os.local_storage`) against actual storage usage in the codebase and the Zepp OS `v3` permission model, convert to the documented v3-safe permission name if required, remove clearly obsolete permissions, and avoid adding speculative permissions for future tasks. Mitigation: if the correct v3 mapping is ambiguous, prefer the smallest documented valid set and log a focused follow-up for Tasks `75` / `76` instead of broadening scope here.
5. Execute subtask `74.4` - validate page module entries: verify the eight existing page files (`page/index`, `page/setup`, `page/game`, `page/summary`, `page/history`, `page/history-detail`, `page/game-settings`, `page/settings`) are each registered exactly once in the v3 manifest structure, keep path names consistent with current navigation code, and ensure no stale module keys remain from the removed `app-side` / `setting` modules.
6. Execute subtask `74.5` - remove legacy properties and finalize cleanup: strip remaining `v2`-only manifest data flagged by the validator (legacy target names, `deviceSource`, obsolete wrapper/layout properties, and any deprecated compatibility fields), retain i18n/default-language fields only if the validated v3 schema still supports them, and avoid unrelated runtime/doc updates in this task. Rollback note: if a field's status is uncertain, keep the last validator-clean form instead of deleting aggressively and document the deferred cleanup.
7. Run the validation sequence in increasing cost order: JSON/schema validation first, then Zepp build validation (`npm run build:all` or equivalent), then simulator smoke on `w390-s`, `w454-r`, `w466-r`, and `w480-r` as far as current migration state allows, and finally `npm run complete-check`. Separate pure manifest failures from later-task runtime failures so Task `74` remains scoped correctly.

## Validation
- Success criteria: `app.json` is accepted as a Zepp OS `v3` manifest, preserves `appId` `1108585` and version `3.0.0`, targets exactly `w390-s`, `w454-r`, `w466-r`, and `w480-r`, retains all eight page registrations, uses a documented v3-safe permission set, removes legacy device-targeting / `v2`-only properties, and passes build plus `npm run complete-check` without manifest-related regressions.
- Checkpoints: Pre-checkpoint confirms repo baseline and schema source before edits; subtask `74.1` checkpoint verifies identity/version/runtime values match current repo truth; subtask `74.2` checkpoint verifies the four screen-family targets fully replace `gtr-3` / `gts-3`; subtask `74.3` checkpoint verifies permissions match actual usage and documented v3 format; subtask `74.4` checkpoint verifies all eight page paths exist on disk and in the manifest; subtask `74.5` checkpoint verifies deprecated `v2` properties are removed or intentionally retained with validator justification; final checkpoint verifies manifest validation, Zepp build acceptance, per-family simulator recognition, and `npm run complete-check`.
