## Task Analysis
- Main objective: Refactor the existing screen adaptation layer so `getScreenMetrics()` and all layout consumers support `w390-s`, `w454-r`, `w466-r`, and `w480-r`, while preserving the current look as much as possible and applying only a minimal fixed top reservation on square devices.
- Identified dependencies: Task `41` established `utils/screen-utils.js`, round insets, layout presets, and layout-engine integration; Task `74` already moved `app.json` to the four screen-family targets and created family-specific asset folders; the current implementation still hardcodes `GTS 3` assumptions in `utils/screen-utils.js` and tests. Relevant modules to inspect first are `utils/screen-utils.js`, `utils/design-tokens.js`, `utils/layout-engine.js`, `utils/layout-presets.js`, `utils/ui-components.js`, `page/index.js`, `page/setup.js`, `page/game/ui-binding.js`, `page/summary.js`, `page/history.js`, `page/history-detail.js`, `page/settings.js`, `page/game-settings.js`, `tests/screen-utils-consolidation.test.js`, `tests/layout-engine.test.js`, `tests/layout-presets.test.js`, `tests/home-screen.test.js`, `tests/game-screen-layout.test.js`, `tests/summary-screen.test.js`, and the family asset folders under `assets/gt.w390-s`, `assets/gt.w454-r`, `assets/gt.w466-r`, and `assets/gt.w480-r`.
- System impact: This is a cross-cutting but layout-scoped refactor touching runtime metrics, token lookup, page schemas, visual sizing, test fixtures, and asset validation; it should not alter scoring, navigation, or persistence behavior, and it must stay compatible with the current Zepp widget-based runtime rather than introducing a redesign or new platform-only UI controls.

## Chosen Approach
- Proposed solution: Replace the current GTS-3-only detection with a small screen-family registry in `utils/screen-utils.js` that classifies the four supported families, returns stable metadata (`screenFamily`, `isRound`, `safeTop`, `statusBarHeight`, and any narrow scaling fields needed by tokens), and keeps `safeTop` as the single layout contract consumed by the existing layout engine; then layer family-aware token selectors on top of the existing token system and only adjust pages where a real overlap or clipping risk remains.
- Justification for simplicity: The codebase already centralizes layout resolution through `getScreenMetrics()`, `resolveLayout()`, and shared presets, so the minimum-safe path is to strengthen those shared abstractions instead of creating per-family page forks; this keeps current layouts intact on `w390-s`, avoids overengineering with four fully separate design systems, and limits custom behavior to the one explicitly confirmed product decision: a fixed square top reservation.
- Components to be modified/created: Modify `utils/screen-utils.js` first; then update `utils/design-tokens.js` and only the downstream helpers that actually consume size assumptions (`utils/ui-components.js`, and only if needed `utils/layout-presets.js` / `utils/layout-engine.js`); audit and patch page schemas in `page/index.js`, `page/setup.js`, `page/game/ui-binding.js`, `page/summary.js`, `page/history.js`, `page/history-detail.js`, `page/settings.js`, and `page/game-settings.js`; extend targeted tests; add an asset audit test or documentation note if the existing family folders are incomplete; update `docs/UI-SYSTEM.md` only if the square status-bar policy is not sufficiently discoverable from utility JSDoc.

## Implementation Steps
1. Confirm the runtime baseline before editing: verify `app.json` still targets exactly `w390-s`, `w454-r`, `w466-r`, and `w480-r`; confirm whether `hmSetting.getDeviceInfo()` exposes family-distinguishing dimensions on this runtime or whether a fallback key is needed; inventory every existing `getScreenMetrics()` consumer and every page that renders a header so the change stays adaptation-only and does not leak into business logic. (Risk check: if runtime dimensions are normalized by `dw`, use a secondary discriminator from runtime device info rather than forcing unreliable width heuristics.)
2. Refactor `utils/screen-utils.js` into a family-first adapter: replace `GTS_3_SQUARE_*` constants and `isGts3SquareDevice()` with a four-family configuration map and a single resolver that returns `screenFamily`, `statusBarHeight`, `safeTop`, `isRound`, and any minimal sizing metadata needed downstream; add `getStatusBarHeight()` as a thin helper; keep `safeTop` backward-compatible so existing layout-engine callers do not need broad rewrites; document the policy directly in JSDoc, explicitly stating that `w390-s` reserves a fixed top inset and round families reserve `0`. (Mitigation: keep old property names working to avoid breaking all page imports at once.)
3. Update `utils/design-tokens.js` with narrow family-aware selectors instead of duplicating the whole token tree: preserve the current shared token ratios as the default baseline, add a helper such as `getFamilyTokens()` or equivalent for the few values that may need family-specific scaling, and route `getFontSize()` / icon sizing through that helper only where the larger round families materially need different values; prefer one shared round profile unless testing proves `w466-r` or `w480-r` need separate overrides. (Rollback note: if family-specific token scaling destabilizes round layouts, keep shared round tokens and scope this task to family detection plus square safe-top support.)
4. Reuse the existing layout pipeline instead of redesigning pages: inspect `utils/layout-engine.js` and `utils/layout-presets.js` to ensure `metrics.safeTop` remains the single source of truth for top reservation, then audit header-bearing pages starting with `page/index.js`, `page/setup.js`, and `page/game/ui-binding.js`, followed by the list/detail/settings pages; replace only the remaining hardcoded sizes or positions that do not naturally scale from metrics/tokens, and preserve current visual composition on `w390-s` except for the fixed top reservation. (High-risk area: the game header and score area are the most sensitive to clipping because they mix percentage layout with explicit icon/text sizes.)
5. Audit asset handling with the v3 screen-family structure in mind: verify the four `assets/gt.*` directories contain the required icon set used by pages, confirm that manifest-based family asset resolution already covers file selection so runtime asset-switching logic is not added unnecessarily, and only add missing files, corrected dimensions, or a lightweight asset-presence test if the audit finds gaps. (Mitigation: keep asset loading by filename if the manifest/folder structure already selects the correct family variant.)
6. Expand automated coverage around the shared adaptation seam: add or update tests in `tests/screen-utils-consolidation.test.js` for all four families and the documented square status-bar policy; extend `tests/layout-engine.test.js` and `tests/layout-presets.test.js` to prove `safeTop` still offsets headers/body correctly; add matrix-style smoke assertions in `tests/home-screen.test.js`, `tests/game-screen-layout.test.js`, and the page tests most likely to clip, using mocked device info for `w390-s`, `w454-r`, `w466-r`, and `w480-r`; add an asset-folder completeness check if no such test exists.
7. Validate in increasing cost order: run the focused unit tests first, then the full `npm run complete-check`, then simulator/manual smoke passes for the acceptance flows (`Start Game`, `Resume Game`, `Scoring`) on all four families; capture any layout regressions by family, and if a regression appears on round families, prefer tightening family-specific token/layout overrides over broad page redesign.

## Validation
- Success criteria: `getScreenMetrics()` deterministically returns the correct family for `w390-s`, `w454-r`, `w466-r`, and `w480-r`; `w390-s` pages reserve the documented fixed top safe area without header overlap; round-device layouts render without clipping in header, score, list, and footer regions; existing flows (`Start Game`, `Resume Game`, `Scoring`) behave the same across families; the family asset folders are complete and the selected assets render at the intended size.
- Checkpoints: Pre-implementation - confirm runtime family detection inputs and that Task `74` assets/manifest are still aligned; during implementation - after Step 2, all metric and safe-top tests pass with the new family map before any page edits; after Step 3, font/icon/token changes are verified on at least one square and one round family; after Step 4, header-bearing pages no longer depend on legacy GTS-3 naming or assumptions; after Step 5, asset audit shows no missing family files or unnecessary runtime switching; post-implementation - targeted tests, `npm run complete-check`, and simulator/manual smoke for `w390-s`, `w454-r`, `w466-r`, and `w480-r` all pass, with rollback limited to token overrides if larger-round tuning introduces regressions.
