## Task Analysis
- Main objective: Establish the Zepp OS 3.6+ app identity baseline by setting a new `appId`, defining a fresh version baseline for the new product line, and making a final cleanup decision on `app-side` and `setting` modules (with user rule: remove if unused).
- Identified dependencies: Task #74 depends on Task #73 for `appId` and version baseline; current manifest is `app.json` v2 with legacy identity (`appId: 1106804`, `version.name: 1.13.0`); version sync tooling (`scripts/sync-version.js`) couples `package.json` -> `app.json` -> `utils/version.js`; `app-side/index.js` and `setting/index.js` are placeholder stubs and currently referenced only from `app.json`; docs/CI references still mention `app-side/` and `setting/` paths.
- System impact: Medium, repo-wide configuration/documentation impact with low runtime logic risk; changes touch manifest identity, module topology, release path filters/docs, and verification flow (JSON validity, reference integrity, build/simulator smoke, QA gate).

## Chosen Approach
- Proposed solution: Use a manifest-first and evidence-based cleanup flow mapped to subtasks 73.1-73.5: (A) set identity/version baseline in `app.json` (and align version-linked files), (B) audit module usage via code search, (C) remove `app-side` and `setting` if audit confirms no runtime need, (D) update docs and release-path references to reflect the decision. Deepthink alternatives rejected: keep stubs "just in case" (adds debt, violates user preference), or delete first and investigate later (unsafe).
- Justification for simplicity: This approach reuses existing project conventions (single-source manifest + documented release policy), avoids speculative architecture work, and makes one deterministic decision path: audit then remove when unused. It minimizes moving parts while still covering CI/docs fallout from directory removal.
- Components to be modified/created: `app.json` (identity fields, module entries), `package.json` + `package-lock.json` + `utils/version.js` (version baseline alignment if reset is approved), `app-side/` and `setting/` (expected deletion), `.github/workflows/release.yml` and `RELEASE.md` (remove stale releasable-path entries if modules are removed), `README.md` (project tree/notes), `CHANGELOG.md` (identity/module decision entry), and this plan file. Dependency/order rationale: apply 73.1 first (unblocks 74), complete 73.2/73.3 audits before 73.4 deletion, then finalize 73.5 docs once outcomes are final.

## Implementation Steps
1. Pre-implementation assumptions and baseline checkpoint (supports all subtasks): confirm expected inputs are present (new Zepp Console `appId`, intended version baseline value, naming policy for new listing), snapshot current manifest/module/doc state, and lock non-goals (no Zepp 3 schema migration here; Task #74 handles full v3 target migration).
2. Subtask 73.1 - Update app identity/version baseline: edit `app.json` identity fields (`app.appId`, and agreed identity metadata), set version baseline (recommended default: `1.0.0` for new app line), and align version-coupled files (`package.json`, `package-lock.json`, `utils/version.js`) so release automation does not reintroduce drift.
3. Subtask 73.2 - Review `app-side` usage and purpose: inspect `app-side/index.js` and i18n payload, run repository-wide reference search excluding docs noise, and record whether any runtime path, import, or lifecycle dependency requires side-service retention.
4. Subtask 73.3 - Review `setting` usage and purpose: inspect `setting/index.js` and i18n payload, verify references separately from `page/settings.js` (to avoid false positives), and document whether the module contributes runtime behavior beyond manifest registration.
5. Subtask 73.4 - Execute module decision (default path per user requirement): if 73.2/73.3 confirm no runtime use (expected), remove `app-side/` and `setting/`, remove their module declarations from each target in `app.json`, and update release path filters/docs that still mention those directories. Elevated-risk mitigation and rollback: if build/runtime validation reveals manifest constraints requiring module presence, immediately restore directories and module entries from git diff, keep minimal stubs, and record explicit retention rationale.
6. Subtask 73.5 - Document identity and module decisions: add a `CHANGELOG.md` entry for new app identity/version baseline and module cleanup decision; update `README.md` (structure/notes) and any migration docs that track bootstrap decisions so Task #74 and later tasks inherit accurate assumptions.
7. Execute verification sequence tied to test strategy: validate `app.json` syntax and structure, run no-broken-reference checks (`grep`/search), perform Zepp build and simulator smoke on square/round targets (launch + home render + basic navigation), then run `npm run complete-check` for final repository QA.
8. Final acceptance and handoff package: capture exact changed files, decision rationale (why modules removed or retained), verification evidence, and explicit blockers (if any) for orchestrator approval before implementation merge.

## Validation
- Success criteria: `app.json` uses the new 3.6+ app identity baseline (`appId` and version baseline agreed), `app-side` and `setting` are removed when confirmed unused, no broken references/imports remain, build/simulator smoke passes, `npm run complete-check` passes, and documentation clearly records identity/module decisions with rationale.
- Checkpoints: Pre-checkpoint (Step 1) validates required inputs and freezes assumptions; during-checkpoints validate each subtask immediately (73.1 manifest/version coherence, 73.2/73.3 audit evidence, 73.4 post-removal reference/build integrity, 73.5 documentation completeness); post-checkpoint confirms full QA and smoke results. Failure handling: if `appId` is unavailable, pause only identity finalization and proceed with review/document prep; if module removal causes runtime/build regression, rollback module deletion and retain stubs with dated rationale; if version baseline conflicts with release automation, keep baseline decision documented and add a follow-up action to align semantic-release policy before first 3.6+ production release.
