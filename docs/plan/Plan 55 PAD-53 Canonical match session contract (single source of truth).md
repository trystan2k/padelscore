## Task Analysis
- Main objective: Deliver Task 55 by defining one canonical match session contract (schema + types + runtime validation + examples + migration docs) that can be reused by storage, history, and UI flows without changing gameplay behavior.
- Identified dependencies: Task 53 decisions and Task 54 storage baseline; current contract code in `utils/match-state-schema.js`; session persistence in `utils/active-session-storage.js`; history mapping in `utils/match-history-types.js`; current Node test harness in `tests/*.test.js`; QA gate `npm run complete-check`.
- System impact: Medium-high on persistence compatibility and schema ownership; elevated regression risk around timestamp format (ISO-8601 vs current numeric fields), status enum expansion (`paused`), and required team identity fields; low direct UI risk if page flow logic is left unchanged.

## Chosen Approach
- Proposed solution: Evolve `utils/match-state-schema.js` into the canonical contract owner (single source), add formal schema/docs in `docs/schema/`, add a TypeScript declaration contract file (without introducing a TS build pipeline), and keep compatibility normalizers for legacy persisted payloads.
- Justification for simplicity: Deepthink compared (A) full TypeScript + JSON-schema library rollout, (B) additive contract evolution in existing JS/JSDoc module, and (C) parallel new contract module with later migration; (B) is simplest and safest because it avoids toolchain/runtime overhead and avoids a temporary dual-contract state.
- Components to be modified/created: Create `docs/schema/match-session.json`; create `docs/schema/match-session.md`; create `utils/match-session-contract.d.ts` (or `types/match-session.d.ts`, chosen to match repo structure); update `utils/match-state-schema.js` (types/validators/serialize/deserialize/migration); create `tests/fixtures/match-session-examples.js`; create `tests/match-session-contract.test.js`; update impacted tests (`tests/match-state-schema.test.js`, `tests/active-session-storage.test.js`, `tests/match-history-storage.test.js`) only where contract changes require it.

## Implementation Steps
1. Phase 0 - Scope lock and assumptions check: confirm dependency Task 53 is complete; lock non-goals (no scoring-rule changes, no page navigation redesign); set assumptions explicitly: keep JS runtime modules, provide TS contract through declaration file, and maintain backward compatibility for existing numeric timestamp usage during transition.
2. Phase 1 - Author canonical JSON Schema in `docs/schema/match-session.json`: define required fields (`teams`, `scores`, `settings`, `status`, `metadata`, `timing`, `setHistory`, `schemaVersion`), enums (`setsToPlay`, `setsNeededToWin`, `status`), numeric bounds, string patterns (matchId), and ISO-8601 `date-time` constraints for timing fields.
3. Phase 2 - Add TypeScript contract aligned to schema: create declaration interfaces/types (`MatchSession`, `MatchTeamIdentity`, `MatchScores`, `MatchSettings`, `SetHistoryEntry`, `MatchStatus`, `SchemaVersion`) and keep naming/file conventions kebab-case; ensure exported types mirror schema keys exactly.
4. Phase 3 - Implement runtime contract functions in `utils/match-state-schema.js`: add/extend `validateMatchSession`, `serializeMatchSession`, `deserializeMatchSession`, and guard helpers; keep existing exports as compatibility wrappers where needed so current imports remain stable.
5. Phase 3 risk control (elevated risk): timestamp/status changes can break migration and resume checks; mitigation is dual-parse normalization (accept legacy numeric timestamps and canonical ISO strings) plus explicit status handling (`active|finished|paused`) while preserving current app behavior for non-active sessions; rollback path is to keep legacy validator gate as fallback behind existing `isMatchState` compatibility path.
6. Phase 4 - Create canonical examples in `tests/fixtures/match-session-examples.js`: add `emptyNewSession`, `activeInProgressSession`, `finishedSessionWithHistory`, and `specialCharacterTeamNamesSession`; ensure UTF-8 examples include accents and emoji and remain schema-valid.
7. Phase 5 - Create schema/migration documentation in `docs/schema/match-session.md`: include field reference table, invariants (for example matchStartTime immutability post-creation), backward compatibility policy (additive-first, deprecation window, no silent removals), and migration workflow for `schemaVersion` increments.
8. Phase 6 - Add/adjust tests: validate pass/fail cases, round-trip serialization, edge numeric bounds, special-character handling, and ISO-8601 parsing; verify example fixtures pass validation; add regression assertions for active-session persistence and history mapping compatibility.
9. Phase 7 - Execute verification gates and finalize: run targeted contract tests first, then regression tests for impacted persistence/history modules, then full project QA gate; if regressions appear, revert only contract-enforcement changes and keep additive docs/schema artifacts while reworking compatibility mappings.
