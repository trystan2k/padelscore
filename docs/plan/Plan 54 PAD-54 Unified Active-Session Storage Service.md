## Task Analysis
- Main objective: Deliver Task 54 by introducing one authoritative active-session persistence service with deterministic APIs (`getActiveSession`, `saveActiveSession`, `clearActiveSession`, `migrateLegacySessions`), UTF-8 file I/O, and legacy migration without breaking current Zepp OS v1.0 flows.
- Identified dependencies: Canonical session contract and validators in `utils/match-state-schema.js`; existing dual persistence paths in `utils/match-storage.js` and `utils/storage.js`; lifecycle entrypoint in `app.js`; session consumers in `page/index.js`, `page/game.js`, `page/setup.js`, and `page/summary.js`; match history service in `utils/match-history-storage.js` and `utils/match-history-types.js`; test filesystem harness in `tests/helpers/hmfs-mock.js`.
- System impact: High on session durability/resume reliability and medium on startup flow; broad regression surface across persistence, resume/start-new-match, and summary-history linkage.

## Chosen Approach
- Proposed solution: Implement a dedicated unified module `utils/active-session-storage.js` as the single write path, keep compatibility wrappers in `utils/match-storage.js` during cutover, perform dual-read migration from legacy sources with idempotent cleanup, and run one safe migration trigger during app startup.
- Justification for simplicity: Deepthink compared (A) big-bang page refactor now, (B) incremental unified service with compatibility wrappers, and (C) generalized storage framework; (B) is the simplest effective option because it satisfies Task 54 fully while minimizing immediate page churn reserved for PAD-55/PAD-56.
- Components to be modified/created: Create `utils/active-session-storage.js` and legacy fixtures under `tests/fixtures/legacy-sessions/`; update `utils/match-storage.js`, `utils/storage.js`, and `app.js` for compatibility/migration wiring; adjust `page/summary.js` only if history compatibility requires metadata pass-through; add `tests/active-session-storage.test.js` and `tests/active-session-storage.integration.test.js`.

## Implementation Steps
1. Establish a pre-implementation contract matrix (Subtask 1) that maps Task 54 API requirements to existing canonical fields in `utils/match-state-schema.js`, explicitly defining non-goals (no scoring/UI behavior changes) and dual-read-only migration policy.
2. Create `utils/active-session-storage.js` API surface and JSDoc types (Subtask 1) with constants for schema version usage and one canonical active-session file path, plus exported methods `getActiveSession`, `saveActiveSession`, `clearActiveSession`, and `migrateLegacySessions`.
3. Implement core storage methods (Subtask 2) using explicit UTF-8 encode/decode and platform-safe hmFS path resolution, ensuring `getActiveSession` returns `null` on missing/corrupt data, `saveActiveSession` serializes through canonical schema helpers, and `clearActiveSession` is idempotent.
4. Implement `migrateLegacySessions` (Subtask 3) to read legacy sources (`utils/storage.js` payloads, previous active-session key/file, and optional in-memory handoff snapshot), normalize into canonical schema, resolve conflicts by newest timestamp, persist once, and clean up legacy artifacts only after successful unified write; include idempotency guard so reruns are no-ops.
5. Wire compatibility and rollout safety (Subtasks 2 and 3): keep `utils/match-storage.js` as a thin adapter over the new service, keep legacy read compatibility but prevent new dual writes, and trigger migration once from `app.js` startup with fail-safe behavior.
6. Verify and enforce match history compatibility (Subtask 4) by preserving fields consumed by `createMatchHistoryEntry` (`status`, `teams`, `setsWon`, `setHistory`, optional completion metadata) and adding an integration seam proving unified-session payloads flow cleanly into Task 29 history storage.
7. Add robust error handling/logging (Subtask 5) with consistent message prefixes and severity (`warn` for write/cleanup failures, `info`/`debug` for migration decisions), while preserving non-throwing behavior for runtime stability.
8. Add comprehensive tests and fixtures (Subtask 6): unit tests for API/type guards, UTF-8 round-trip (including special characters/emojis), read/write error paths, migration conflict resolution and idempotency, and integration tests for startup migration + history compatibility using legacy fixture files.
9. Run quality gates and regression verification (Subtasks 5 and 6): execute `npm run complete-check`, then perform manual resume/start-new-match/history smoke checks (mid-game exit/reopen, migration from legacy file, no duplicate legacy artifacts), documenting any rollout caveats.

## Validation
- Success criteria: Unified service is the only active-session write path; all four API methods behave per Task 54 contract; UTF-8 data round-trips without corruption; migration selects newest legacy state and cleans legacy stores idempotently; history integration remains compatible with Task 29 structures; error handling returns safe defaults and emits actionable logs; automated tests and `npm run complete-check` pass.
- Checkpoints: Pre-implementation checkpoint confirms contract matrix and migration non-goals before coding; during-implementation checkpoint A validates core API and UTF-8 behavior before wiring migration; checkpoint B validates migration conflict resolution, cleanup idempotency, and rollback trigger (disable startup migration and keep legacy read fallback) before broad integration; checkpoint C validates history compatibility and logging semantics; post-implementation checkpoint runs full automated QA plus manual resume/history regression scenarios.
