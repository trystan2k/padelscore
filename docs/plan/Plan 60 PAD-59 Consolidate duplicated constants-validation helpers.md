## Task Analysis
- Main objective: Consolidate duplicated constants, validation helpers, and small utility helpers across `page/*`, `app.js`, and `utils/*` into shared modules (`utils/constants.js`, `utils/validation.js`) with zero behavioral change and strict Task 60 scope.
- Identified dependencies: Task 53 canonical contract and validation ownership in `docs/schema/match-session.json`, `docs/schema/match-session.md`, and `utils/match-state-schema.js`; duplicated helper call sites in `page/game.js`, `page/index.js`, `page/summary.js`, `page/setup.js`, `page/score-view-model.js`, `page/history.js`, `page/history-detail.js`, `app.js`, `utils/storage.js`, `utils/active-session-storage.js`, `utils/match-history-storage.js`, `utils/scoring-engine.js`, and `utils/match-session-init.js`.
- System impact: Medium-high because duplicated helpers are used in gameplay resume/state normalization and persistence flows; regression risk is highest in score conversion, set-history normalization, date formatting, and storage fallback constants.
- Subtask mapping: 60.1 -> Steps 1-2; 60.2 -> Step 3; 60.3 -> Step 4; 60.4 -> Steps 5-6; 60.5 -> Steps 7-8.

## Chosen Approach
- Proposed solution: Use an audit-first, family-by-family consolidation: (A) inventory and classify duplicates (exact vs subtle differences), (B) add shared `utils/constants.js` and `utils/validation.js` with canonical implementations plus behavior-preserving variants/options where needed, (C) migrate imports incrementally per duplicate family with immediate verification after each family.
- Justification for simplicity: Deepthink comparison favored this over (1) a big-bang replacement of all duplicates in one commit (high blast radius) and (2) introducing new abstraction layers/repositories (overengineered for this task). Incremental consolidation reuses existing architecture, keeps review diff readable, and minimizes behavior-drift risk.
- Components to be modified/created: Create `utils/constants.js` and `utils/validation.js`; update duplicate helper/constant call sites in `page/game.js`, `page/index.js`, `page/summary.js`, `page/setup.js`, `page/score-view-model.js`, `page/history.js`, `page/history-detail.js`, `app.js`, `utils/storage.js`, `utils/active-session-storage.js`, `utils/match-history-storage.js`, `utils/scoring-engine.js`, and `utils/match-session-init.js`; update/add focused tests in `tests/setup-flow.test.js`, `tests/home-screen.test.js`, `tests/game-screen-layout.test.js`, `tests/summary-screen.test.js`, `tests/start-new-match-flow.test.js`, `tests/match-storage.test.js`, `tests/active-session-storage.test.js`, `tests/match-session-init.test.js`, and `tests/unification-regression/*.test.js` as needed.

## Implementation Steps
1. (60.1) Build a duplicate audit matrix (file, symbol/constant, current behavior/signature, call sites, risk level) using `rg` sweeps for known duplicate families: `isRecord`, `toNonNegativeInteger`, `toPositiveInteger`, `cloneMatchState`, `cloneSetHistory`/`normalizeSetHistory`, `resolveWinnerTeam`/`isTeamIdentifier`, `isTieBreakMode`, persisted point mapping (`50/60`), set-option constants (`1/3/5`), `formatDate`, and hmFS fallback flags (`FS_O_*`); record subtle differences explicitly (for example `cloneSetHistory` fallback rules and sorted vs unsorted history behavior).
2. (60.1) Establish a pre-change baseline and assumptions checkpoint: run `npm test` and `npm run test:unification`, capture pass/fail baseline, and freeze non-goals (no optimization/refactor outside consolidation, no behavior changes, and no forced unification where semantics differ by call site).
3. (60.2) Create shared module structure: add `utils/constants.js` for canonical constants and `utils/validation.js` for shared validators/small helper functions; prefer deriving contract-linked values from existing canonical sources (for example `SETS_TO_PLAY`, `SETS_NEEDED_TO_WIN`, `MATCH_STATUS`, `SCORE_POINTS`) instead of re-declaring literals.
4. (60.3) Implement canonical helper families in shared modules with backward-compatible signatures and behavior-preserving variants/options for subtle differences; for non-identical duplicates, keep per-call-site behavior via explicit options or thin wrappers (for example set-history fallback index behavior, sorted vs unsorted output) and document each decision in Task 60 audit notes.
5. (60.4) Migrate duplicate usage to imports in controlled slices (lowest risk first): (a) date formatting duplication (`page/history.js`, `page/history-detail.js`), (b) generic validators (`isRecord`, integer helpers, team-id checks), (c) persisted score conversion/tie-break constants and helpers (`page/index.js`, `page/game.js`), (d) set-option constants/validation (`page/setup.js`, `utils/match-session-init.js`), (e) storage fallback constants/helpers (`utils/storage.js`, `utils/active-session-storage.js`, `utils/match-history-storage.js`) where signatures are identical.
6. (60.4) After each migration slice, run targeted checks for that slice before proceeding; rollback/mitigation rule for elevated-risk slices: revert only the last migrated duplicate family (not the whole task), restore previous local helper for that family, then re-introduce consolidation with tighter compatibility wrappers.
7. (60.5) Execute full verification and duplication scans: run targeted tests for touched modules, then `npm run complete-check`; run duplicate-search commands excluding canonical shared modules to ensure no audited duplicate implementations remain outside approved exceptions.
8. (60.5) Final acceptance pass: confirm canonical contract alignment with Task 53 (status/timestamp/set-config validations still enforced), confirm subtle differences are documented, and produce a final checklist in task notes showing each audited duplicate family is consolidated or explicitly retained with rationale.
