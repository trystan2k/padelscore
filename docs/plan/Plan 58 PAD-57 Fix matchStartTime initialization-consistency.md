## Task Analysis
- Main objective: Implement PAD-57 so match start timestamp behavior is deterministic, canonical, and immutable after initialization (except explicit migration/repair), while preserving Zepp OS v1.0 runtime compatibility and existing storage flows.
- Identified dependencies: Task 53 canonical contract ownership in `utils/match-state-schema.js` + `docs/schema/match-session.json`/`docs/schema/match-session.md`; Task 54 unified storage APIs in `utils/active-session-storage.js`; session creation/persistence entrypoints in `utils/match-session-init.js`, `utils/match-storage.js`, `page/setup.js`, and `page/game.js`; coverage baseline in `tests/match-state-schema.test.js`, `tests/match-storage.test.js`, `tests/match-session-init.test.js`, `tests/active-session-storage.test.js`, and `tests/active-session-storage.integration.test.js`.
- System impact: Medium-high on migration and persistence correctness (timestamp semantics across create/save/load/update/migrate), medium on page session-start flows, and medium regression risk for resume/history behavior if timestamp normalization changes are not tightly scoped.

## Chosen Approach
- Proposed solution: Treat `matchStartTime` as the canonical start-time invariant represented by `timing.startedAt` (existing schema field), then enforce it end-to-end via contract clarification, storage-level normalization/immutability guards, deterministic session creation initialization, and explicit migration/repair paths for legacy sessions missing start time.
- Justification for simplicity: Deepthink comparison favored this over two alternatives: (A) adding a brand-new top-level `matchStartTime` field + schema-version bump (high churn, duplicate source of truth), and (B) introducing a standalone migration subsystem/script framework (overengineered for current scope). Reusing the existing canonical timing model is the smallest change set that satisfies the task requirements.
- Components to be modified/created: `utils/match-state-schema.js` (symbols: `normalizeTiming`, `createDefaultTiming`, `buildNormalizedMatchState`, `tryNormalizeMatchState`, `isCanonicalMatchState`); `utils/active-session-storage.js` (symbols: `saveActiveSession`, `updateActiveSession`, `updateActiveSessionPartial`, `normalizeLegacyRuntimeState`, `migrateLegacySessions`, candidate selection helpers); `utils/match-session-init.js` (`initializeMatchState`); `utils/match-storage.js` (`saveMatchState` normalization behavior); `page/setup.js` (`handleStartMatch`, `persistMatchStateForGameStart`); `page/game.js` (`createPersistedMatchStateSnapshot` fallback base shape); `docs/schema/match-session.md` (explicit invariant text) and optionally `docs/schema/match-session.json` only if contract wording/validation must be tightened; test updates in `tests/match-session-init.test.js`, `tests/match-storage.test.js`, `tests/match-state-schema.test.js`, `tests/active-session-storage.test.js`, `tests/active-session-storage.integration.test.js`; if symbol locations drift, run discovery with `rg -n "matchStartTime|startedAt|timing|initializeMatchState|saveActiveSession|updateActiveSession|migrateLegacySessions" utils page tests docs/schema`.

## Implementation Steps
1. Subtask 58.1 - Audit and contract lock: produce a code-path matrix for every create/save/load/update/migrate read/write of start-time semantics; document that `matchStartTime` maps to canonical `timing.startedAt`, identify all mutation paths currently able to overwrite it, and confirm schema/docs gaps (especially immutability wording and migration policy wording).
2. Subtask 58.1 - Pre-change assumptions checkpoint: verify accepted policy defaults before edits (derivation priority uses earliest reliable session timestamp when available, fallback to `created_at` semantics mapped to canonical `timing.createdAt`; immutability exception scope limited to explicit migration/repair options only).
3. Subtask 58.2 - Storage handling hardening: update unified storage write/update pipeline to preserve existing `timing.startedAt` once present, ensure missing start time is filled deterministically at first canonicalization, and prevent accidental overwrite in `saveActiveSession` and update helpers unless an explicit migration/repair flag is supplied.
4. Subtask 58.3 - Migration policy implementation: add a dedicated migration/repair normalization path for sessions missing start time that derives the value using ordered candidates (earliest reliable timestamp source list agreed in Step 2), then fallback to `timing.createdAt`; make migration idempotent and traceable through existing migration flow (`migrateLegacySessions` and deserialization normalization).
5. Subtask 58.4 - Deterministic session creation flow: ensure all session creation entry points stamp start time exactly once at initialization (`initializeMatchState` primary path, plus `page/game.js` fallback snapshot path when no canonical base exists) before downstream state mutations/persistence.
6. Subtask 58.5 - Immutability guards: add centralized guard logic at contract/storage boundary so attempts to change initialized start time are rejected or ignored with explicit logging; allow override only through explicit migration/repair code paths to avoid silent drift from gameplay update flows.
7. Subtask 58.5 - Test expansion: add focused unit tests for initialization timing order, immutability enforcement, migration derivation precedence/fallback, and storage round-trip preservation; add integration coverage for startup migration + save/load/update cycles to verify start time stays stable after first initialization.
8. Elevated-risk mitigation and rollback gate: if timestamp regressions appear in resume/history flows, rollback only start-time guard changes at call sites first (keep schema docs/tests and migration fixtures), then reintroduce in smaller slices behind one explicit repair-option pathway; do not roll back unrelated Task 54 storage unification behavior.
9. Final verification: run targeted node tests for touched modules first, then `npm run complete-check`; execute a manual smoke path (setup -> game -> scoring updates -> app destroy/reopen -> summary/history save) to confirm start-time consistency and no lifecycle regressions.

## Validation
- Success criteria: `timing.startedAt` is initialized deterministically at session creation, remains unchanged across normal save/update operations, can only be changed by explicit migration/repair path, legacy sessions missing start time are repaired using earliest reliable timestamp then `created_at` fallback mapping, schema/docs clearly define this invariant, and all targeted plus full QA checks pass (`npm run complete-check`).
- Checkpoints: Pre-implementation assumptions check after Steps 1-2 (contract mapping and derivation policy confirmed); during-implementation correctness check A after Step 3 (storage save/update cannot mutate initialized start time); check B after Step 4 (migration derivation order and idempotency validated with fixtures); check C after Steps 5-7 (all creation entry points initialize once and guard behavior is test-covered); post-implementation regression check after Step 9 (targeted tests + full QA + manual lifecycle smoke); risk list with mitigations: R1 contract-name drift (`matchStartTime` vs `timing.startedAt`) mitigated by explicit doc mapping and test assertions, R2 legacy payload ambiguity mitigated by deterministic candidate priority + fallback, R3 false-positive immutability blocking valid repairs mitigated by explicit migration/repair override option and rollback note in Step 8.
